## 2. 불변 제약 & 가드레일

1. **사실 정확성(안전 직결) — 최우선.** 거리·소요시간·난이도·표고·들머리·위험구간·교통은 **권위 소스로 조사하고 ≥2개 교차검증**한다. 추정·창작 금지. 불확실 항목은 본문에 `확인 필요`로 명시하고 단정하지 않는다.
2. **템플릿 정합성.** 신규 플레이북은 **기존 통합 플레이북(post-rebrand)** 과 구조·클래스·디자인시스템(`assets/css/design-system.css`)·컴포넌트(탭/코스/체크포인트표/고도/팁/경고/갤러리/교통/히어로 영상)를 100% 동일하게 따른다. 새 마크업 발명 금지 — 가장 가까운 기존 산을 **스켈레톤으로 복제 후 내용만 교체**.
3. **빌드리스·다크/라이트·a11y(AA)·반응형** 유지. 데스크톱 2단 split·검색/필터 연동.
4. **출처 기록.** 모든 사실 출처를 `_orchestration/research-<id>.json`과 `CREDITS.md`에 기록.
5. **자산 일관.** 히어로 영상은 기존 실사 Ken Burns 방식(≤2MB), OG 카드는 산별 합성 카드. AI 생성물 없음.
6. **무회귀·원자적 커밋.** 산 1개=커밋/PR 단위. main 직접 푸시 금지. 검증 GREEN 전 머지 금지.
7. **index 등록.** 완성 시 `MOUNTAINS[]`에 `done:true` + `url` 설정, 카드 스탯(거리/시간/난이도)도 연구값과 일치.

## 3. 토폴로지 & DAG

```
                 ┌─────────────────────────────┐
                 │  ORCHESTRATOR (리드)          │
                 └──────────────┬──────────────┘
       산 단위 병렬(5)           ▼
   ┌───────────────────────────────────────────────────┐
   │  per-mountain:  A-RESEARCH ─► A-BUILD ─► A-ASSET    │
   └───────────────────────────────────────────────────┘
                                ▼
                     ┌────────────────────┐   FAIL  ┌──────────────┐
                     │ A-VERIFY (적대적)   │ ──────► │ 원인 단계 반송 │
                     └─────────┬──────────┘         └──────────────┘
                               │ GREEN
                               ▼  index 등록 → 페이지 PR
```

| Phase | 담당 | 게이트 |
|---|---|---|
| P1 리서치 | A-RESEARCH (산별 병렬) | `research-<id>.json` 완비 + 핵심수치 ≥2소스 교차검증 |
| P2 빌드 | A-BUILD | 기존 템플릿 복제·내용 주입, 로컬 렌더 OK |
| P3 자산 | A-ASSET | hero.mp4(≤2MB)·og.png·poster 생성, 영상 연결 |
| P4 검증 | A-VERIFY (독립) | 사실·템플릿·a11y·링크 모두 PASS |
| P5 등록 | 오케스트레이터 | index MOUNTAINS[] done:true+url, baekhaksan 버그 해소 |

**파일럿 우선**: 무등산(국립공원·3코스, 데이터 풍부) 1산을 끝까지 완성→사용자 확인 후 나머지 4산 병렬 확장.

## 5. 에이전트 프롬프트 (복붙용)

> 공통 헤더: §2 제약 주입. 특히 "사실 정확성 최우선, 추정 금지, 불확실은 '확인 필요'".

### A-RESEARCH (산별 병렬)
```
역할: 등산 정보 리서처. 배정된 산 1개의 코스·체크포인트·교통·계절·위험요소를 조사한다.
소스 우선순위: ① 국립공원공단(knps.or.kr)·해당 국립공원 공식 ② 산림청·두루누비·지자체 관광 ③ 보조(블로그는 교차검증용, 단독 근거 금지).
작업:
1) 산 개요(표고·정상명·지정현황), 대표 코스 2~3개(난이도별)와 각 코스의 거리/소요/체크포인트/표고/위험구간.
2) 대중교통 접근(시내버스·역·들머리 접근), 케이블카 등 시설.
3) 계절 하이라이트·주의(결빙·낙석·통제구간), 실용 팁.
4) 모든 핵심 수치는 ≥2개 권위 소스로 교차검증, 출처 URL 기록. 편차는 범위로, 불명은 '확인 필요'.
산출물: _orchestration/research-<id>.json (§4 스키마).
금지: 수치·코스 창작, 단일 블로그 단독 근거, 출처 없는 단정.
```

### A-BUILD (산별)
```
역할: 플레이북 빌더. research-<id>.json으로 기존 통합 템플릿에 맞는 <id>-playbook.html을 만든다.
선행: 구조가 가장 유사한 기존 플레이북을 스켈레톤으로 선택해 복제
  (국립공원·3코스형 → odaesan/sobaeksan류, 소규모·단일코스형 → sikjangsan류). 실제 파일을 정독해 클래스/구조 파악.
작업:
1) 스켈레톤 복제 후 모든 콘텐츠를 research 값으로 교체(코스·체크포인트표·고도·팁·경고·교통 섹션).
2) 히어로: 기존 실사 사진(assets/img/<id>/hero-*) + 디자인시스템 컴포넌트, 데스크톱 2단 split.
3) SEO/OG/twitter 메타, alt(한/영), 사진 크레딧 캡션(기존 CREDITS 기준).
4) 본문 수치는 research와 1:1 일치. '확인 필요' 항목은 그대로 표기.
산출물: <id>-playbook.html (브랜치 커밋).
금지: 디자인시스템 외 임의 스타일, research에 없는 사실 추가, 다른 산 클래스 가정.
```

### A-ASSET (산별)
```
역할: 자산 생성기. <id>의 히어로 영상·OG 카드 생성·연결.
작업:
1) 기존 실사 Ken Burns ffmpeg 파이프라인으로 assets/video/<id>/hero.mp4(720p·~12s·≤2MB)+poster(실사 재사용).
2) 산별 OG 카드(1200x630, 실사 히어로+산이름+Korea Trails) 생성 → assets/img/<id>/og.png, 메타 연결.
3) hero-bg-video 패턴(데스크톱 한정·muted/loop/playsinline·preload=none·lazy·reduced-motion 정적 폴백) 주입.
4) asset-manifest.json·CREDITS.md 갱신(ai_generated:false, 실사 출처).
산출물: 영상·OG·poster + 매니페스트/크레딧.
완료기준: 영상 ≤2MB, 데스크톱 자동재생·모바일 폴백 동작.
```

### A-VERIFY (독립 적대적)
```
역할: 검증관. 생산자 편을 들지 않는다.
1) 사실: 본문 수치(거리·시간·난이도·표고·교통)가 research-<id>.json 및 출처와 일치하는가. 출처 없는 단정·창작 의심 적발. ≥2소스 교차검증 여부.
2) 안전: 위험구간·통제·계절 주의가 누락 없이 반영됐는가. 단정 과잉 없는가.
3) 템플릿: 기존 통합 플레이북과 구조·컴포넌트·디자인시스템 일치(스타일 드리프트 0).
4) a11y/SEO: 대비 AA·키보드·alt·메타, Lighthouse 유지.
5) 링크/무회귀: index 카드→플레이북 연결, 영상·이미지 200, 콘솔 에러 0.
판정: FAIL 1건이라도 RED → 원인 에이전트 반송(최대 2라운드). ALL GREEN만 등록·머지.
```

## 7. 마스터 킥오프 프롬프트 (오케스트레이터에게 복붙)

```

권위 소스로 조사하고 그 결과로 기존 통합 템플릿에 맞는 플레이북을 만들어 완성한다.
MISSING-MOUNTAINS-RESEARCH.md를 단일 진실원천으로 삼는다.

규칙:
1) 모든 서브에이전트 상단에 §2 제약 주입. 최우선은 '등산 정보 사실 정확성'(안전 직결):
   추정·창작 금지, 핵심 수치는 권위 소스 ≥2개 교차검증, 불확실은 '확인 필요'로 표기.

먼저 파일럿으로 무등산 1개를 끝까지(리서치→빌드→자산→검증→등록) 완성해 사용자 확인을 받은 뒤,

리서치 핵심 수치는 출처와 함께 제시하라. 시작 전 전체 실행계획을 요약해 승인받아라.
```
